SiReSe: Simple Remote Security

A simple way to remotely deploy applications into Secure Elements

There are millions of tamper-resistant microchips all over the world. These chips, called "Secure Elements", are secure execution environments for applications, having a very strict security architecture enforced with help of a cryptoprocessor and lots of permissions and keys.
Examples of such Secure Elements are the chips in some credit cards or the SIM cards in most mobile phones. They are the modern evolution of Smart Cards, and are typically programmed and personalized in dedicated facilities.
Due to the rollout of mobile-NFC, many smartphones now have a very user-friendly interface for the applications running in their Secure Elements: they can be used in all readers being rolled out for contactless cards: payments, transit, loyalty, access control...
a great opportunity, but there is always something, in this case:

The issue

Deploying applications to Secure Elements in mobile phones is extremely hard. Exotic standards from the Mobile world and from the SmartCard closed tradition, along with security considerations, mean an expensive implementation, followed by long negotiations with whoever controls the Secure Element: mobile operators, smartphone manufacturers and other industry players all set their own rules.
Each of them sets its own technical connectivity interfaces and rules, pricing and contractual models, so a company wishing to deploy its applications to its customer's phones will need to set up, and maintain over time, many connections with a huge technical and business overhead.
This is one of the main barriers delaying, or even stopping, extension of mobile NFC services.

Our solution

SiReSe's goal is to break this situation: a service that will abstract all this complexity, connecting a Service Provider directly to its customers' phones!
We are developing a cloud service, accesible through a webservices API, to hide most of the complexity in mobile NFC:

    SiReSe's platform will connect to companies controlling Secure Elements, implementing the interfaces and processes required by each one of them, and signing all the necessary contracts and agreements.
    Our API operations are just the ones needed to deploy and manage applications. SiReSe's platform deals with the nitty- gritty details of managing Secure Elements' architecture

How to get there

Implementing the platform and API is a long, hard process. The team behind SiReSe brings experience in the mobile security industry: having worked in a big company in the field, we know very well why is it so hard to make mobile-NFC work, and we think we can bring some barriers now. Our short-term goal is to code the core components of the SiReSe platform, so they can be used to easily put together some demos and build proof-of-concept systems with application developers, to validate our core assumptions. For that goal, we have calculated we need $100,000, which we plan to invest in hiring a few developers and to accelerate the process of building these core components.
Once we reach this short-term goal, the next step will be to launch a VC funding round, in order to beef up the team with sales engineers so we can start connecting to Secure Element owners, focusing mostly in mobile operators in North America and in Europe.

FAQ

What is a Secure Element?
It is a tamper-resistant chip, containing an execution environment, crypto co-processor, RAM and flash-memory.
Can you tell me some examples of applications running on Secure Elements?
    EMV credit cards, called 'chip and pin' in the UK
    Cards to access public transit systems
    Access control cards, like the employees badges used to enter facilities
    Cards used to open some kind of rental cars, such as those of Hertz on Demand
What programming languages are used for applications in Secure Elements?
Mostly a subset of Java called JavaCard.
